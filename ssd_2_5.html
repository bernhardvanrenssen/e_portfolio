Collaborative Discussion 1: Summary Post

Summary Post

As defined by OWASP (2021), Cross-site scripting is an attack that occurs when:

  • Data is entered in a web application from an untrusted source, often a malicious web request.
  • This data is posted as dynamic content, and sent through to the user without being validated for any malicious content.

As discussed in my Initial post, this often happens in the form of a segment of JavaScript (but could also be some form of HTML – or any other code that a browser can execute). Because web applications rely heavy on user-input, it becomes an easy target for attackers performing malicious activities through these input areas. In my initial post I also highlighted several mitigations that a developer can implement to reduce or (hopefully) negate any such malicious script injections.

As I did not have a lot of knowledge prior to this module on security and security vulnerabilities, I did some research on how easy it is to inject scripting on exiting websites. I found an XSS – vulnerability website (https://xss-game.appspot.com/) where I could test out some XSS attacks with various levels of difficulty. Besides being a fun activity, it indeed showcased how easy it is to execute such an attack on poorly developed web applications, by simply using existing input fields.

After reading a peer response by Kei Yui Yvone Chan, I was pointed towards OWASPs Enterprise Security API (ESAPI) that is quite a handy and up-to-date set of tools to prevent any XSS attacks. As a team, I will propose this method as a way to validate input fields in our design application for this module.
