What is ReDOS and what part do ‘Evil Regex’ play?

ReDOS is an acronym for Regular Exporession Denial-of-Service attack. Regular Expressions is simply a string of text that allows a user to create patterns that can help match or locate that pattern in a body of text – basically a search/lookup function (Computer Hope, 2020) A ReDOS is thus an attack that causes a denial-of-service – meaning that the system will freeze or take a very long time to respond, due to it trying to execute the Regex command entered, causing any user to not be able to use the system at all whilst the system tries to resolve the regex command or lookup (Weidman, n.d.). An ‘Evil Regex’ is a Regex command that an attacker carefully crafted, that will cause this ReDOS attack. With the special syntax that Regex uses to validate a user’s input, these carefully constructed inputs can exploit the way that the lookup is handled, by taking the longest available route to locate/handle the input string – for example in a username/password input box.

What are the common problems associated with the use of regex? How can these be mitigated?

Common problems with regex include that it few expressions fail to compile. This is a problem because the syntax used in most regex is sensitive to the placement of the character in its relation to the characters it needs to lookup. Regular expressions thus need to be always correct, or it could easily return false information, instead of failing. The user will be unaware that the actual regex is wrong and will continue thinking that the correct lookup is being returned (No exception is being thrown by the code). A common way to mitigate these regex risks, is by using tools that detect uncommon or irregular use of regex, that might be due to the programmer coding incorrect regular expressions. One such tool -ACRE (Automatic Checking of Regular Expressions) is described by Larson (2018) as “…a tool that detects mistakes in regular expressions”. The tool works by allowing the user to insert the desired regex and clicking the “Check” button. It then runs a series of 11 checks (Larson, 2018), focusing on mistakes that the user might have entered and the part that contains the irregularity/error is then highlighted, for the user to inspect and correct.

How and why could regex be used as part of a security solution?

There are a couple of reason how and why developers would use regular expressions as a tool to improve or secure their web applications. Regex allows the developer to (Li, 2020):
a. Create rule blocks for Firewalls to filter through certain file types or IP addresses that might contain malicious content.
b. Do user input validation. This is a very common use for Regex, as it will validate user input, ensuring that no malicious content can be entered – for example characters that can alter a request made to the system, commonly used in a SQL-injection or XSS (Cross Site Scripting) attack.
c. Detect possible dangerous content in files, so that malware can quarantine files accordingly.
